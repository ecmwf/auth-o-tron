# Migrating from HMAC JWTs

Auth-O-Tron 0.4.0 and `authotron-client` 0.2.0 make a breaking change to JWT issuance and verification. `authotron-types` is also 0.2.0 because `AuthError` gains the public `InvalidPublicKey` variant. Auth-O-Tron now signs and verifies RS256 tokens only. There is no HS256 compatibility mode.

## Server configuration

Both configuration schema versions use the same JWT settings. This means the change applies to version `1.0.0` and version `2.0.0` files. Automatic conversion of the version 1 server layout only maps `bind_address`; it does not migrate `jwt.secret`.

Replace the old settings:

```yaml
jwt:
  iss: auth-o-tron.example.com
  aud: my-application # optional in the previous configuration
  exp: 3600
  secret: legacy-shared-secret
```

with:

```yaml
jwt:
  iss: auth-o-tron.example.com
  aud: my-application
  exp: 3600
  kid: key-2026-01
  private_key: set-via-AOT_JWT__PRIVATE_KEY
```

Generate a key pair and inject the private key from a secret:

```bash
openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:3072 -out jwt-private.pem
openssl pkey -in jwt-private.pem -pubout -out jwt-public.pem
export AOT_JWT__PRIVATE_KEY="$(cat jwt-private.pem)"
```

RSA keys below 2048 bits are rejected at startup. New keys should be 3072 bits. `iss`, `aud`, `kid`, and `private_key` are required in both configuration versions.

## Rust client API

The previous API accepted one shared HMAC secret and returned `AuthClient` directly:

```rust,ignore
let client = AuthClient::new(url, secret, timeout, cache_ttl, cache_capacity);
let user = decode_jwt(token, secret)?;
```

The new API accepts a public keyset identified by `kid`, requires issuer and audience, and returns `Result` because keys are parsed during construction:

```rust,ignore
use authotron_client::{AuthClient, JwtPublicKey, decode_jwt};

let public_keys = [JwtPublicKey::new(
    "key-2026-01",
    include_bytes!("jwt-public.pem"),
)];

let client = AuthClient::new(
    url,
    &public_keys,
    "auth-o-tron.example.com",
    "my-application",
    timeout,
    cache_ttl,
    cache_capacity,
)?;
let user = decode_jwt(
    token,
    &public_keys,
    "auth-o-tron.example.com",
    "my-application",
)?;
```

Consumers must pin RS256 and reject tokens with missing or unknown `kid` values. The supplied client also rejects duplicate `kid` values and every RSA public key below 2048 bits in its configured keyset.

## Deployment order for this breaking migration

HS256 and RS256 tokens are deliberately not accepted by the same verifier. Upgrade the server configuration and every consumer as one coordinated release, allowing for a brief authentication interruption if they cannot be switched atomically. Do not expose the private key to consumers.

After the initial migration, future RSA key changes can be zero-downtime. Follow the staged overlap procedure in [JWT Signing](../configuration/jwt.md): publish the new public key first, switch the signer, wait the maximum token lifetime, then retire the old public key.
